# Research: Row/Card Differentiation, Selected States, and Compact Input Controls (M3 + Flutter)

This document is primary-source research only — it contains no implementation decisions. It exists
to answer the research question opened by
[github.com/xDeZex/universal/issues/211](https://github.com/xDeZex/universal/issues/211), which
blocks Prototype tickets #212 (row/card/divider/selection visual language) and #213 (input control
redesign) under the "Gym-tracking visual design spec" map (#210). Every claim below cites the
primary source it came from: official Material Design 3 documentation (`m3.material.io`), official
Flutter API docs (`api.flutter.dev`), or first-party app resources. Where no first-party source
could be found, that is stated explicitly rather than filled in from secondary blog write-ups.

**Fetching note**: `m3.material.io` is a client-rendered Angular SPA — a plain HTTP fetch of these
URLs returns only an empty shell (`<mio-root></mio-root>` plus script tags), confirmed by direct
`curl`. The prose quoted below was retrieved via a JS-rendering fetch of the same canonical URLs
(the rendered DOM of the pages cited, not a third-party summary or mirror), and every quote is
attributed to its `m3.material.io` URL of origin.

## (a) Differentiating rows/cards without relying solely on a divider

**The core M3 answer: don't use a divider as the primary separator for selectable/contained lists — use a gap between individually-shaped, individually-colored item containers instead.** A divider is a single shared line that a full-bleed selection highlight will always be able to paint over (because the highlight and the divider occupy the same z-position, edge-to-edge). A gap is empty space belonging to neither row, so there's nothing for a highlight to erase.

- M3's list guidelines state this directly: "Gaps or dividers can separate lists into items and groups: Use **gaps** for contained lists. Gaps leverage expressive shape and containment tactics. Limit **dividers** to uncontained or complex lists, only when a stronger visual separation is necessary." A "Caution" callout on the same page reinforces: "Limit the use of **dividers** to uncontained lists." ([m3.material.io/components/lists/guidelines](https://m3.material.io/components/lists/guidelines), "Gaps & dividers" section)
- The lists *specs* page makes this a named variant distinction, not just a tip: **"Expressive lists"** ("recommended for new designs") have "a segmented style and round corners," while **"Baseline lists"** ("original M3 component designs... don't have the latest visual style, selection treatment, and slot functionality") have "square corners and standard colors." The baseline variant is the one with an actual `Divider` element in its anatomy and its own divider-measurement table (100% full-width, 16dp inset-left/24dp inset-right padding, 0dp padding above/below); the expressive variant's spec has no divider row at all — separation there comes from the segmented/rounded-corner container itself. ([m3.material.io/components/lists/specs](https://m3.material.io/components/lists/specs), "Variants" and "List (baseline)" sections)
- The same guidelines page shows this pattern by name for contained groups: "Use **segmented gaps** and filled list items to define a list group" (image caption, "Gaps & dividers" section) — i.e. each row is its own filled, separately-shaped container; the "list" is really a stack of small cards with breathing room between them.
- Cards guidelines describe the sibling pattern for card collections: "By default, cards in a collection are coplanar. They share the same resting elevation unless they're picked up or dragged." ([m3.material.io/components/cards/guidelines](https://m3.material.io/components/cards/guidelines), "Cards in a collection"). Coplanar cards are inherently gap-separated (margin between card containers), never divider-separated.
- Cards *do* use an internal divider, but only **within** one card's content, and the choice of full-width vs. inset divider is itself meaningful: "Use full-width dividers for content that can be expanded" vs. "Use inset dividers, which don't run the full width of a card, to separate related content." ([m3.material.io/components/cards/guidelines](https://m3.material.io/components/cards/guidelines), "Dividers" section, under Anatomy). This is a divider *inside* a single selectable/expandable unit, not a divider *between* two independently-selectable units — so it never has to compete with a selection highlight.

**Actionable pattern for #212**: stop using a shared `Divider` between `PlannedExerciseCard`/row items. Give each row its own container (rounded corners, e.g. via `ListTile.shape` or `Card.shape` — see Flutter mapping below) and separate rows with a gap (`SizedBox`/spacing in a `Column`, or `ListView.separated` with a spacer instead of a `Divider`) rather than a line. This is exactly the M3 "expressive list" pattern, not a workaround invented for this project.

### Flutter mapping
- `ListTile.shape` is documented as: "Defines the tile's `InkWell.customBorder` and `Ink.decoration` shape" — settable independently per tile, so each row can carry its own rounded-rectangle shape rather than sharing a flat, dividerless strip. ([api.flutter.dev/flutter/material/ListTile-class.html](https://api.flutter.dev/flutter/material/ListTile-class.html))
- `Card` itself "is a Material Design card: a panel with slightly rounded corners and an elevation shadow," and exposes `shape` ("The shape of the card's Material"), `color`, `margin`, and `elevation` independently per instance ([api.flutter.dev/flutter/material/Card-class.html](https://api.flutter.dev/flutter/material/Card-class.html)) — i.e. `margin` is the Flutter equivalent of an M3 "gap."
- `Divider`'s own doc frames it as line-based separation for flat/uncontained lists: "A thin horizontal line, with padding on either side... Dividers can be used in lists, Drawers, and elsewhere to separate content," with a dedicated helper `ListTile.divideTiles` "optimized for this case" ([api.flutter.dev/flutter/material/Divider-class.html](https://api.flutter.dev/flutter/material/Divider-class.html)) — confirming `Divider` is the baseline-list tool, not the expressive/contained-list tool.

## (b) Selected/expanded states that coexist with dividers instead of erasing them

**The core M3 answer: selection is a solid container-color swap on the row's own background (a discrete color-role change), not a translucent state-layer overlay painted over existing content.** Because it's the row's *own* fill color, it sits *underneath* anything that belongs to that row (its internal divider, its rounded shape) rather than as a wash on top of the whole list.

- **Selection is a color/shape change, not an opacity layer.** M3's Selection foundation page: "Selection is shown through changes to surface color or other visible elements... Selections are displayed using a check mark icon, a checkbox component, a change in surface color, or a combination." ([m3.material.io/foundations/interaction/selection](https://m3.material.io/foundations/interaction/selection), overview bullets and "Selection indicators")
- **State layers (hover/focus/press/drag) are a *separate*, quantified mechanism from selection**, and their exact values are: "Four overlay states and their values: Hover +8% opacity, Focus +10% opacity, Press +10% opacity, Drag +16% opacity." The mechanism: "The state layer is an overlay with a fixed opacity for each state and uses the same color as the content... if the enabled style uses the secondary container color for the container and on secondary container for content, the state layer will be an overlay using the on secondary container color." Size note: "The size of state layers is 40dp while the interactive target size is 48dp." ([m3.material.io/foundations/interaction/states/state-layers](https://m3.material.io/foundations/interaction/states/state-layers)) — notably, **selected/activated are absent from this four-item opacity list**, which corroborates that "selected" is handled by the color-role system, not the opacity-overlay system.
- **The concrete color token for a selected list item**: the expressive list's color-roles diagram lists 10 roles including "Primary container" / "On primary container" as the dedicated selected-item pairing, distinct from the default item's "Surface" / "On surface(-variant)" roles, and the specs page has explicit separate "States" subsections for "Default list items" (Enabled, Disabled, Hovered, Focused, Pressed, Dragged) vs. "Selected list items" (the same six states, but starting from the selected container color, not the default one). ([m3.material.io/components/lists/specs](https://m3.material.io/components/lists/specs), "Color" and "States" sections)
- **This selected-container approach composes with the gap-based separation from (a) automatically**: since the selected color is scoped to a single row's own container (which already has its own rounded shape and its own margin from neighbors), selecting a row can never visually erase a boundary — there is no shared line to erase in the first place.
- **Where M3 does use dividers alongside a stateful/expandable element, it prescribes which divider style to use for which purpose**: "Use full-width dividers for content that can be expanded" (cards guidelines, "Dividers" section, cited fully in (a) above) — i.e. if a divider must coexist with an expand interaction, M3's own answer is "make it full-width and put it at the seam between the always-visible header and the expandable body," not "let a highlight overpaint a divider that spans the whole row."
- **List items can expand in place**, and this is a first-party-documented interaction, not something this project would be inventing: the lists specs page's interaction-configuration table lists "Interactions: Expand — Available" for both M3 and M3 Expressive lists ([m3.material.io/components/lists/specs](https://m3.material.io/components/lists/specs), "Configurations" → "List interactions" table), and separately "List items containing other list items can expand and collapse in a folder-like manner, to reveal or hide content. Tapping a list item expands it vertically across the entire screen using a container transform transition pattern." ([m3.material.io/components/lists/guidelines](https://m3.material.io/components/lists/guidelines), "List interactions" → "Expand & collapse") — note the guidelines text describes the canonical example as expanding *to fill the screen* (a `container transform` transition), which is a stronger claim than "expands inline in place"; cards guidelines describe the identical pattern for cards ("Cards can use a container transform transition pattern to reveal additional content. Reserve this pattern for hero moments that are meant to be expressive," with an explicit "Don't scroll within a card to reveal information"). **Gap/caveat**: M3's own canonical example of "expand" is a full-screen parent-child transition, not an in-place/inline row expansion that keeps its neighbors' positions intact (which is what this repo's Planned Exercise row editor currently does, per `universal/lib/widgets/planned_exercise_row_editor.dart`). No M3 primary source found describing an inline/in-place expansion pattern explicitly — treat "expand this row in place without navigating away" as an adaptation of the documented pattern, not a literal instance of it, when writing #213/#212's spec.
- **Card elevation as a state signal is reserved for drag, not selection**: "Some components, such as list items, chips, or cards, can inherit elevation to signify a dragged state" ([m3.material.io/foundations/interaction/states/applying-states](https://m3.material.io/foundations/interaction/states/applying-states), "Dragged" section) and cards guidelines: "When moving a card, increase its elevation" ("Pick up & move" section). Selected state, per the Selection page and the lists color-roles diagram above, is communicated by color/shape, not by elevation — so a selected row shouldn't be implemented by lifting its shadow; that channel is reserved for drag semantics.

### Flutter mapping
- `ListTile` has exactly this container-color-swap built in: `selectedTileColor` — "Defines the background color of ListTile when selected is true" — paired with `tileColor` ("...when selected is false") and `selectedColor` ("Defines the color used for icons and text when the list tile is selected"), all independent of the tile's `shape`. Caveat from the same doc: these colors "are not painted by the ListTile itself but by the Material widget ancestor," so `ListTile` needs a `Material` ancestor (typically the `Scaffold`) for `selectedTileColor` to animate. ([api.flutter.dev/flutter/material/ListTile-class.html](https://api.flutter.dev/flutter/material/ListTile-class.html))
- `Card.color`, `Card.elevation`, and `Card.shape` are independently settable per instance, so a "selected" card can swap `color` (e.g. to a `colorScheme.primaryContainer`-style role) without touching `elevation` — matching the M3 finding above that selection ≠ elevation change. ([api.flutter.dev/flutter/material/Card-class.html](https://api.flutter.dev/flutter/material/Card-class.html))

## (c) Compact numeric input controls (steppers, unit toggles) as a coherent unit

**The core M3 answer for a two-value toggle (kg/lb, reps/time): use a single-select segmented control, not two independent toggle buttons.** For a stepper (-/+ around a number): pair two of M3's own icon-button *style* variants (filled/filled-tonal/outlined), which are explicitly designed to be combined and to read at a graded emphasis level — not bare unstyled "+/-" text.

### Segmented buttons / connected button groups (unit toggles)

- M3's canonical example for a single-select segmented control is a size/unit picker: "Use a single-select segmented button to select one option from a set, switch between views, or sort elements from up to five options. For example, use a single-select segmented button to choose one of a set of sizes, such as this beverage size selector." ([m3.material.io/components/segmented-buttons/guidelines](https://m3.material.io/components/segmented-buttons/guidelines), "Single-select" section) — structurally identical to a kg/lb or reps/time toggle.
- Container: "Like common buttons, segmented buttons have fully rounded corners by default" (same page, "Container" section); measurements: height 40dp, outline width 1dp, min 12dp left/right padding, 8dp padding between elements, 48dp touch target, with "each step down in density remov[ing] 4dp from the height" ([m3.material.io/components/segmented-buttons/specs](https://m3.material.io/components/segmented-buttons/specs), "Measurements" / "Density"). Selected-segment color roles: on-surface/outline for unselected, secondary-container/on-secondary-container for selected (same page, "Color").
- **Important currency flag**: M3's button-groups documentation states plainly, in normal body copy (not just a tooltip), that segmented buttons are being superseded: "**Connected button groups** ... should replace the baseline segmented button ... which is no longer recommended" and "Use instead of segmented button ... which is no longer recommended" ([m3.material.io/components/button-groups/guidelines](https://m3.material.io/components/button-groups/guidelines), "Usage"; [m3.material.io/components/button-groups/overview](https://m3.material.io/components/button-groups/overview), "Availability & resources" area). The site's own inline glossary tooltip for "Segmented buttons," which recurs verbatim on both pages, reads: "Segmented buttons help people select options, switch views, or sort elements. Note: They're deprecated in the expressive update. Use a nav rail instead." The "use a nav rail instead" clause reads as a mismatched/copy-pasted tooltip (a nav rail is a navigation-destination component, not a substitute for a two-way value toggle) and should not be taken literally — but the surrounding plain-prose claim ("no longer recommended," "should replace the baseline segmented button") is corroborated in two independent places and should be treated as the current, real M3 Expressive guidance.
- **Practical resolution for this repo**: Flutter's `SegmentedButton` class ("A Material button that allows the user to select from limited set of options... typically used in cases where there are only 2-5 options" — [api.flutter.dev/flutter/material/SegmentedButton-class.html](https://api.flutter.dev/flutter/material/SegmentedButton-class.html)) carries **no deprecation notice** as of this research, and Flutter has no `ButtonGroup`/"connected button group" widget to replace it with. So `SegmentedButton` remains the only first-party Flutter widget for this job; #213 should use it, but style it toward the "connected button group" visual language M3 now favors (fully rounded, selected-segment shape change) rather than treating the M2-era boxy segmented look as current — see the M2-vs-M3 comparison M3 itself draws: "M2: Segmented buttons had a small corner radius and label text in all caps" vs. the current "fully rounded corners" and sentence-case labels ([m3.material.io/components/segmented-buttons/overview](https://m3.material.io/components/segmented-buttons/overview), "Differences from M2").
- Selected-state visual language for a *connected button group* (if a future Flutter widget adopts it, or if this is hand-rolled from `IconButton`/`FilledButton` primitives): "When a button in a standard group is selected: The selected button changes shape and width. A selected toggle button also changes color. Adjacent buttons move and temporarily change width" and "A selected button should change shape from round to square, or square to round" ([m3.material.io/components/button-groups/guidelines](https://m3.material.io/components/button-groups/guidelines), "Usage" and "Behavior" → "Selected") — i.e. selection is communicated by *shape change*, not color alone, echoing the icon-button toggle rule below.

### Icon buttons (stepper -/+ controls)

- M3 defines four icon-button emphasis styles in one ranked scale: "Filled, Tonal, Outlined, Standard... For the highest emphasis, use the filled style. For the lowest emphasis, use standard." Specific guidance directly applicable to a stepper: "Use the **tonal** style as a middle ground between filled and outlined icon buttons. It's useful for secondary actions paired with a high emphasis action," and "Use the **outlined** style for medium-emphasis buttons." ([m3.material.io/components/icon-buttons/guidelines](https://m3.material.io/components/icon-buttons/guidelines), "Usage")
- Icon buttons are explicitly designed to be placed adjacent to each other with shared interaction behavior: "When placed in a button group, adjacent icon buttons respond to one another when pressed" and "Multiple icon buttons can be placed in a standard button group to add interaction and motion between the buttons when pressed" (same page, intro + "Usage"). This is the first-party basis for treating a -/+ pair as one coherent grouped control rather than two independent buttons floating next to a number.
- Toggle-style icon buttons (relevant if a stepper's buttons also need an on/off/pressed identity) must change more than color: "Toggle buttons should use an outlined icon when unselected, and a filled version of the icon when selected... This is to ensure that selection is communicated through at least two properties, rather than just color" (same page, states section).

### Flutter mapping
- Flutter's `IconButton` ships the exact filled/tonal/outlined vocabulary as named constructors: "Material Design 3 introduced new types (standard and contained) of `IconButton`s. The default `IconButton` is the standard type. To create a filled icon button, use `IconButton.filled`; to create a filled tonal icon button, use `IconButton.filledTonal`; to create a[n] outlined icon button, use `IconButton.outlined`." ([api.flutter.dev/flutter/material/IconButton-class.html](https://api.flutter.dev/flutter/material/IconButton-class.html)) — a -/+ stepper pair built from two `IconButton.filledTonal` (medium emphasis, as M3 recommends for "secondary actions paired with a high emphasis action") flanking the numeric value directly satisfies the M3 guidance above, and is a one-line swap from bare `IconButton`/text-button "+/-".
- `FilterChip` ("Filter chips use tags or descriptive words as a way to filter content... a good alternative to Checkbox or Switch widgets") and `InputChip` ("Input chips can be made selectable by setting `onSelected`... They have a label, and they can have a leading icon... and a trailing icon") are the Flutter chip variants most relevant if a unit toggle is rendered as chips rather than a segmented button ([api.flutter.dev/flutter/material/FilterChip-class.html](https://api.flutter.dev/flutter/material/FilterChip-class.html), [api.flutter.dev/flutter/material/InputChip-class.html](https://api.flutter.dev/flutter/material/InputChip-class.html)) — but per the segmented-buttons guidelines above, M3 itself prefers a segmented control over chips specifically when there are only 2 (or up to 5) mutually-exclusive options, reserving chips for filtering/larger option sets: "Use for simple choices between two to five items (for more items or complex choices, use chips)" ([m3.material.io/components/segmented-buttons/overview](https://m3.material.io/components/segmented-buttons/overview)).

### Fitness-app patterns (Strong, Hevy, StrongLifts) — sourcing gap, stated explicitly

No first-party design documentation, design-system write-up, or component-level design rationale was found from Strong, Hevy, or StrongLifts. Specifically:

- A search for Strong's own site (`strongapp.io`) surfaced only its **user-facing help center** (feature how-tos: templates, PRO subscription, integrations) — no developer/design documentation of any kind.
- Hevy's own blog post "How We Built Hevy" ([hevyapp.com/how-we-built-hevy](https://www.hevyapp.com/how-we-built-hevy/)) is a general founder narrative. It states only that the team "planned the entire UI on Sketch and made Marvel prototypes... In addition, I also prepared a library of reusable components, a text & color style guide, and an icon library" — it does not describe any specific component's design (steppers, chips, dividers, etc.), so it provides no usable rationale for this research question.
- No first-party StrongLifts design material was located.

Per the sourcing rules for this research, this sub-question is intentionally left thinner than (a)/(b): no fitness-app-specific patterns are asserted here, and none should be attributed to Strong/Hevy/StrongLifts without a genuine first-party citation. If visual inspiration from these apps is wanted for #213, that would need to come from directly inspecting the shipped apps (a different, non-documentation-based research method) rather than from anything citable as "documented design rationale."
