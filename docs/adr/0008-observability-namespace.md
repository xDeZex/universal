# Shared `observability` namespace for the observability stack

otel-collector, the metrics backend (VictoriaMetrics or Prometheus), Grafana, Loki, and later a traces backend, all share a single `observability` namespace rather than the per-component pattern ADR 0005 uses for infra, or a namespace each. We decided this because ADR 0005's justification for isolation — a component holding elevated privilege whose blast radius should be bounded (Sealed Secrets can decrypt any SealedSecret it watches) — doesn't apply to any of these: none holds RBAC reaching beyond what it's explicitly pointed at. They're also mutually interdependent by design (the collector feeds the backend, Grafana reads the backend and Loki, a traces backend would be another thing Grafana reads), so per-component namespaces would add Kubernetes-level trust boundaries between components that have no actual trust boundary today, with an onboarding tax on every new signal or backend swap (adding a traces backend, switching VictoriaMetrics for Prometheus) and no containment benefit. Keeping them in one namespace still isolates the stack as a whole from `services` (ADR 0004) and from the per-component infra namespaces (ADR 0005). Revisit if a component here ever needs privilege the others shouldn't share.

## Considered options

- **Per-component namespace**, mirroring ADR 0005 — rejected: ADR 0005's trigger is elevated privilege with cluster-wide reach; nothing in the observability stack has that, so there's no trust boundary between these components to enforce, only operational overhead.
- **Shared `infra` namespace with all other infra components** — rejected: would mix observability's read access to cluster-wide metrics/logs with components like Sealed Secrets that hold decrypt privileges — the two categories shouldn't share a blast radius even if they're each internally homogeneous.
