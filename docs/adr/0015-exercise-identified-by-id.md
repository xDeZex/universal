# Exercise identified by id, not name

`Exercise` is the first domain entity in the app referenced from other entities (`Exercise Entry` references it) rather than owned outright, unlike `ChecklistItem`, which is identified by its own name because nothing outside its parent `Checklist` ever points at it. The obvious, consistent choice was to follow the same name-as-identity pattern. Instead, `Exercise` carries a stable `id`, caller-supplied rather than generated by the model itself (deferred to whatever future storage/catalog layer creates Exercises). This lets a user correct an Exercise's freeform name later (e.g. fixing a typo) without orphaning references from existing Exercise Entries — a scenario name-as-identity cannot support without a delete-and-recreate that would disconnect historical Sets.
